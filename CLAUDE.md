# ESP32S3 Camera Project — Claude Context

## Hardware
- Board: Seeed XIAO ESP32S3 Sense
- Camera: OV2640 (built-in)
- PIR: Grove PIR motion sensor — red → 3V3, black → GND, yellow → D0 (GPIO1)
- Camera IP: 10.10.10.155 (streaming firmware) / esp32cam.local (hybrid + deepsleep)
- WiFi networks: SenSen2 / wongabongamcdonga, Home2.4g / 18lookoutway, Gobind's iPhone / 12345678

## Firmware environments

Three firmwares, one board. Upload with `pio run -e <name> --target upload`.

| Environment | File | Behaviour | Best for |
|-------------|------|-----------|----------|
| `streaming` | `src/main.cpp` | Always on, MJPEG + RTSP, web UI, PIR capture | Testing, development |
| `deepsleep` | `src/main_deepsleep.cpp` | Sleeps at ~14µA, PIR wake → 1 photo → sleep | Ultra-low power, 1 shot |
| `hybrid` | `src/main_hybrid.cpp` | Sleeps at ~14µA, PIR wake → 30s MJPEG stream → sleep | Deployment on battery |

## Key firmware concepts (streaming)
- PIR edge detection in `loop()` — rising edge captures JPEG, falling edge schedules baseline update
- In-frame detection: motion JPEG must be >10% larger AND >5KB larger than quiet-scene baseline
- Baseline: EMA (70% old + 30% new), updated 2s after PIR falls quiet
- Ring buffer: last 3 captures in `motionSlots[3]`, protected by `motionMutex`
- RTSP runs as FreeRTOS task on core 0, HTTP handlers spawned per-connection on core 1
- Known bug: no PIR cooldown — one walk-past can produce 2–3 duplicate captures (fix: add 5s cooldown in loop())

## Key firmware concepts (hybrid + deepsleep)
- Deep sleep kills USB — no serial monitor while sleeping; power-cycle to catch boot output
- Power bank keepalive: 30s timer wakeup prevents auto-shutoff (14µA draw too low for power bank)
- mDNS: both firmwares advertise `esp32cam.local` — no need to hunt for DHCP IP
- Hybrid streams at `esp32cam.local/stream` for 30s after PIR fires, then sleeps
- Deepsleep serves one photo at `esp32cam.local/motion` for 30s after PIR fires, then sleeps

## Streaming firmware endpoints
- `/` — live stream + controls (resolution, quality, FPS cap)
- `/stream` — raw MJPEG stream
- `/motions` — gallery of last 3 PIR-triggered captures
- `/stats` — CPU, temp, RAM, FPS, RSSI, motion history
- `rtsp://10.10.10.155:8554/mjpeg/1` — RTSP stream (VLC compatible)

## File structure

```
src/
  main.cpp              — streaming firmware (always-on MJPEG + RTSP)
  main_deepsleep.cpp    — deep sleep firmware (1 photo per PIR wake)
  main_hybrid.cpp       — hybrid firmware (30s stream per PIR wake) ← use for deployment
practice/
  pir_practice.cpp      — scratch/learning firmware, not compiled
  deep_sleep_motion.cpp — experimental deep sleep with in-frame detection
  ESP32_Camera_Knowledge.docx — full knowledge doc written during development
  ESP32_PIR_Notes.docx        — PIR wiring and behaviour notes
  make_knowledge_doc.py — regenerates ESP32_Camera_Knowledge.docx
  make_notes_doc.py     — regenerates ESP32_PIR_Notes.docx
test/
  run_tests.py          — full automated suite (streams, RTSP, stability, endpoints)
  res_fps_test.py       — FPS at each resolution, saves frames to test/frames/
  wifi_test.py          — RSSI + FPS at 1 and 2 viewers for a given location
  motion_test.py        — PIR motion capture test (streaming/hybrid firmware)
  motion_test_deepsleep.py — PIR motion capture test (deepsleep firmware)
  make_test_doc.py      — regenerates test/Cam_tests.docx from hardcoded results
  Cam_tests.docx        — full test results document
  test_results.json     — auto-generated by run_tests.py (gitignored)
  frames/               — sample JPEGs from res_fps_test.py (gitignored)
  motion_captures/      — frames from motion_test.py (gitignored)
  motion_captures_deepsleep/ — photos from motion_test_deepsleep.py (gitignored)
```

## Setup (first time on a new machine)

### 1. Install VS Code + PlatformIO
VS Code → Extensions → search "PlatformIO IDE" → Install

### 2. Clone and open
```bash
git clone https://github.com/Gobnd/esp32-camera.git
cd esp32-camera
```
File → Open Folder in VS Code

### 3. Python deps
```bash
pip install requests tabulate python-docx
```

### 4. Upload firmware
```bash
pio run -e hybrid --target upload        # battery deployment
pio run -e streaming --target upload     # full features / testing
pio run -e deepsleep --target upload     # ultra-low power

pio device monitor --baud 115200         # serial monitor
```

## Running tests

All test scripts run from the project root:

```bash
python3 test/run_tests.py                          # full automated suite
python3 test/res_fps_test.py                       # FPS per resolution
python3 test/wifi_test.py "next to router"         # WiFi range at a location
python3 test/motion_test.py                        # motion capture (streaming/hybrid)
python3 test/motion_test_deepsleep.py              # motion capture (deepsleep)
python3 test/make_test_doc.py                      # regenerate Cam_tests.docx
```

## Test results

### Resolution vs FPS (streaming firmware, clean run 2026-05-21)
| Resolution | FPS | Usable (>20fps)? |
|------------|-----|-----------------|
| QVGA 320×240 | 26.0 | Yes |
| VGA 640×480 | 20.9 | Yes — recommended |
| SVGA 800×600 | 26.5 | Yes |
| HD 1280×720 | 15.2 | Borderline |
| SXGA 1280×1024 | 11.9 | No |
| UXGA 1600×1200 | 8.3 | No |

No hard cliff at VGA - SVGA is also fast. Real drop is at HD and above. VGA still recommended because SVGA gives no extra face recognition distance. Previous 13fps SVGA result was a test artifact (active stream during test). No crashes at any resolution when tested cleanly.

### WiFi range (SenSen2)

Usable streaming (>10fps/viewer) cuts off at the building wall.

### Stability (streaming firmware)
- 300s soak, 3 simultaneous clients, 0 drops, temp 56–64°C — PASS
